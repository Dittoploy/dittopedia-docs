# Tests E2E — Playwright

## Présentation

Les tests end-to-end (E2E) vérifient le comportement de l'application Dittopedia
du point de vue de l'utilisateur final. Ils sont exécutés via
[Playwright](https://playwright.dev/) dans un repository dédié : `dittopedia-playwright`.

## Prérequis

- **Bun** ≥ 1.0
- Le front-end (`dittopedia-front`) accessible sur `http://localhost:3001`
- Le back-end (`dittopedia-back`) accessible sur `http://localhost:3000`

> Les deux services peuvent être démarrés via `docker compose up` depuis `dittopedia-infra/`.

## Installation

```bash
cd dittopedia-playwright

# Installer les dépendances
bun install

# Installer les navigateurs Playwright
bunx playwright install --with-deps
```

## Lancer les tests

```bash
# Tous les tests (headless)
bun test

# Mode visible (headed)
bun run test:headed

# Interface graphique Playwright
bun run test:ui

# Mode debug (step-by-step)
bun run test:debug

# Ouvrir le dernier rapport HTML
bun run test:report
```

### Page Object Model

Les Page Objects encapsulent les sélecteurs et interactions d'une page dans une
classe dédiée. Cela facilite la maintenance et la lisibilité des tests.

```typescript
// pages/home.page.ts
export class HomePage {
  readonly page: Page;
  readonly heading: Locator;

  constructor(page: Page) {
    this.page = page;
    this.heading = page.getByRole('heading', { name: 'Dittopedia' });
  }

  async goto() {
    await this.page.goto('/');
  }
}
```

### Fixtures

Les fixtures Playwright injectent automatiquement les Page Objects dans les tests :

```typescript
// fixtures/base.fixture.ts
export const test = base.extend<DittopediaFixtures>({
  homePage: async ({ page }, use) => {
    const homePage = new HomePage(page);
    await use(homePage);
  },
});
```

### Écrire un test

```typescript
import { test, expect } from '../../fixtures/base.fixture';

test.describe('Home page', () => {
  test('should display the heading', async ({ homePage }) => {
    await homePage.goto();
    await expect(homePage.heading).toBeVisible();
  });
});
```

## Comportement en CI

Lorsque la variable `CI` est définie, la configuration adapte automatiquement :

- **Retries** : 2 tentatives en cas d'échec
- **Workers** : 1 seul worker (exécution séquentielle)
- **Reporter** : `blob` (pour agrégation dans le pipeline)
- **`forbidOnly`** : empêche les `.only` d'être commités
